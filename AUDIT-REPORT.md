# thepopebot v1.2.82 — Security & Reliability Audit Report

## Executive Summary

This is a static review of the `thepopebot` npm package source (event handler, agent-job orchestration, CLI/setup). The overall risk posture is **high**: the package exposes multiple **unauthenticated and cross-tenant remote-code-execution and secret-disclosure paths** in its core request/dispatch surface, compounded by systemic input-validation gaps (shell command building, path traversal, SSRF) and a multi-user authorization model that is enforced inconsistently.

**Counts by severity (confirmed + needs-review):**

| Severity | Count |
|---|---|
| P0 | 5 (3 distinct root causes) |
| P1 | 27 |
| P2 | 64 |
| P3 | 60 |

(Several P0/P1 entries are duplicate reports of the same root cause across subsystems; they are merged below.)

**The most important things to fix first:**

1. **Trigger fan-out runs before webhook-secret verification on public routes** (`api/index.js:484-491`). `fireTriggers()` executes on every POST — including the unauthenticated `/telegram/webhook` and `/github/webhook` public routes — *before* the route handler validates the webhook secret. Any internet caller can fire `command`/`agent`/`webhook` trigger actions with fully attacker-controlled body. **This is the single highest-impact bug.**

2. **`command`-type actions run template-substituted request data through `/bin/sh -c`** (`lib/actions.js:16-18` + `lib/triggers.js:20`). Untrusted `{{body.*}}` values are interpolated unescaped into `child_process.exec`. Chained with #1 this is unauthenticated RCE on the event-handler host (which holds the Docker socket and all settings-DB secrets).

3. **Container-to-host RCE via shell-interpolated untracked filename** (`lib/code/actions.js:624`). `grep -c '' "${file}"` interpolates a `git ls-files`-sourced filename; a file named `$(...)` executes on the host during the 30s diff-stats poll. Reachable by the (untrusted) coding agent.

4. **Client-controlled `scope` path traversal escapes the workspace sandbox** (`lib/ai/scope.js:12`). `path.join(repoRoot, scope)` with no `..` guard lets any authenticated non-admin set the in-process Claude Code SDK `cwd` to the host filesystem (e.g. the settings DB, `.env`), with write/execute if `codeModeType:'code'`.

5. **Pervasive cross-tenant container & secret access** — the `/stream/containers*` SSE endpoints and several server actions (`stopDockerContainer`/`removeDockerContainer`, `getOAuthSecretCredentials`, `getWorkspace`, cluster webhook, `getSessionLog`) check only `requireAuth()` (any logged-in user) where they need `requireAdmin()` or an ownership check. Any non-admin can enumerate/force-remove arbitrary containers, read OAuth client secrets, and read other tenants' workspaces/logs.

6. **Single global encryption key with hardcoded salt and no rotation path** (`lib/db/crypto.js:7-24`). Every secret-at-rest is encrypted under one PBKDF2(AUTH_SECRET, `'thepopebot-config-v1'`) key; obtaining `AUTH_SECRET` decrypts the entire store, and rotation silently mass-degrades all secrets.

## Architecture Overview

- **External /api surface** (`api/index.js`) — catch-all GET/POST dispatcher; `x-api-key` for general callers, per-subsystem webhook secrets for Telegram/GitHub. **Trust boundary:** `PUBLIC_ROUTES` bypass `x-api-key`; trigger fan-out runs on every POST before route auth.
- **Action dispatch + scheduling** (`lib/actions.js`, `cron.js`, `triggers.js`, `maintenance.js`) — three action types (`agent`/`command`/`webhook`); `command`→shell, `webhook`→fetch. **Trust boundary:** user-editable CRONS.json/TRIGGERS.json + `{{body.*}}` request templating feed the executor.
- **AI / LLM integration** (`lib/ai/*`) — chat stream forked between in-process Claude SDK adapter and headless Docker container. **Trust boundary:** client `scope`, `codeModeType`, `chatId`, `repo`/`branch` flow into cwd/permission-mode/workspace selection.
- **Auth** (`lib/auth/*`) — NextAuth v5 JWT sessions; Edge middleware gates `/admin`; `requireAuth`/`requireAdmin` server-side. **Trust boundary:** JWT role is never re-checked against DB; `/api/*` uses separate x-api-key.
- **Channels** (`lib/channels/*`, Telegram) — normalizes webhooks into the chat pipeline; verify-code account binding. **Trust boundary:** Telegram secret (`!==`, non-timing-safe) is the only gate; verify code is 32-bit.
- **Chat server** (`lib/chat/api.js`, `actions.js`) — route handlers (reads) + server actions (mutations). **Trust boundary:** several reads use server actions (convention violation) and several handlers omit ownership checks.
- **Chat / settings UI** (`lib/chat/components/*`) — React client; some admin-only surfaces (secrets) mounted outside `/admin` and gated by `requireAuth` reads.
- **Cluster workspaces** (`lib/cluster/*`) — role-driven Docker workers with cron/webhook/file-watch triggers. **Trust boundary:** cluster webhook accepts any valid key; `getSessionLog` path-joins client segments.
- **Code workspaces** (`lib/code/*`) — interactive containers via ttyd + WS proxy; host-side `execSync` git for diffs. **Trust boundary:** filenames/branches/paths flow into host and in-container shells; forwarded ports exposed via Traefik with no auth.
- **Containers + Docker tools** (`lib/tools/docker.js`, `lib/containers/*`) — sole Docker-socket gateway + SSE status/logs. **Trust boundary:** container `name`/`cleanup` query params unscoped; `resolveHostPath` caches failures.
- **External tool integrations** (`lib/tools/*`, `lib/github-api.js`) — GitHub/Telegram/AssemblyAI/OAuth wrappers + action executor. **Trust boundary:** webhook URLs/headers and template tokens from config files.
- **Database layer** (`lib/db/*`) — SQLite/Drizzle; single multiplexed `settings` table; AES-256-GCM at rest. **Trust boundary:** delete-by-id with no type guard; unguarded JSON.parse/decrypt on rows.
- **Config / instrumentation / utils / oauth / voice** (`config/`, `lib/utils/render-md.js`, `lib/oauth/*`, `lib/voice/*`) — startup wiring, config resolution, markdown includes, OAuth state, voice tokens. **Trust boundary:** `{{include}}` path traversal; OAuth state has no nonce/expiry; OAuth `tokenUrl` unvalidated (SSRF).
- **Web app (Next.js)** (`web/*`) — browser SPA + custom server + WS proxy. **Trust boundary:** WS-proxy parses cookie by hand; `/admin/*` relies solely on middleware.
- **CLI + setup wizard** (`bin/`, `setup/`) — local operator toolchain; scaffolds templates, distributes secrets. **Trust boundary:** CLI args / repo URLs / PAT interpolated into `execSync` shells; `.env` writes unescaped.

**Primary trust boundaries:** (a) internet → `/api/*` (webhook secrets / x-api-key); (b) authenticated non-admin user → admin/cross-tenant resources (session role); (c) untrusted agent-job container → event-handler host (agent-job API key + Docker socket); (d) attacker `AUTH_SECRET` exposure → entire secret store.

## Findings

### [P0] Unauthenticated trigger fan-out before webhook-secret verification → RCE/SSRF (api-external, tools-integrations) — api/index.js:484-491
**Status:** confirmed
**Category:** security
**Description:** In `POST()`, `checkAuth()` returns null (pass-through) for `PUBLIC_ROUTES` (`/telegram/webhook`, `/github/webhook`, `api/index.js:25`), then `fireTriggers(routePath, body, query, headers)` runs **unconditionally** at lines 484-491 — *before* the route handler runs. The GitHub (`handleGithubWebhook`, line 345, `safeCompare`) and Telegram (`telegram.js:36`) webhook-secret checks happen *inside* those handlers, i.e. after fan-out has already fired. A trigger whose `watch_path` matches a public route fires on a fully unauthenticated request with attacker-controlled body/query/headers. (Reported redundantly as P0 and P1 across `api-external` and `tools-integrations`; same root cause.)
**Impact:** Any internet caller who can reach `POST /api/telegram/webhook` or `/api/github/webhook` (no secret, no key) fires matching triggers. Combined with `command` actions (RCE), `webhook` actions (SSRF), or `agent` actions (Docker container launch / LLM spend) this is unauthenticated remote action execution. Even on key-gated routes, the design fires triggers before the secret is validated, defeating the secret as an auth gate.
**Recommendation:** Do not run `fireTriggers` on `PUBLIC_ROUTES`, and never before the route's own auth completes. Either gate with `if (!PUBLIC_ROUTES.includes(routePath))`, or invoke triggers only after each webhook handler verifies its secret. Additionally allowlist which `watch_path`s triggers may bind to so they cannot shadow auth-bearing endpoints.

### [P0] Command injection via template-substituted request data through `child_process.exec` (actions-scheduling, tools-integrations) — lib/actions.js:16-18
**Status:** confirmed
**Category:** security
**Description:** `executeAction()` for `type === 'command'` runs `await execAsync(action.command, { cwd })` where `execAsync = promisify(child_process.exec)` (`/bin/sh -c`). `action.command` is produced by `resolveTemplate()` (`lib/triggers.js:34`), which substitutes `{{body.field}}`/`{{query.field}}`/`{{headers.field}}` with raw request values (`String(data[field])`, `triggers.js:20`) — no escaping/quoting. Any shell metacharacter (`;`, `|`, `$(...)`, backticks, newline) breaks out. Cron jobs feed the same executor. (Reported P0 and P1 across two subsystems; same root cause.)
**Impact:** Arbitrary shell execution on the event-handler host (which holds the Docker socket and all settings-DB secrets) for anyone who can shape a watched POST body — unauthenticated when chained with the public-route ordering bug above, otherwise any valid API key (including the low-privilege agent_job_api_key).
**Recommendation:** Do not pass templated request data into a shell. For `command` actions use `execFile`/`spawn` with an argv array (no shell) and pass substituted values as discrete args; or disallow `{{...}}` tokens in `command`, or strictly shell-quote every substituted value at load time.

### [P0] Host RCE via shell-interpolated untracked filename in `grep -c` (code-workspaces) — lib/code/actions.js:624
**Status:** confirmed
**Category:** security
**Description:** `getWorkspaceDiffStats()` runs host-side `execSync` against the workspace clone, enumerating untracked files with `git ls-files --others --exclude-standard` and then for each running `` execSync(`grep -c '' "${file}"`, opts) `` (line 624), interpolating the raw filename inside double quotes. `git ls-files` emits printable-ASCII names verbatim; inside `sh` double quotes, `$(...)` and backticks still execute. Confirmed locally that a file named `$(touch HOSTPWNED).txt` runs the command. The code page polls `/code/workspace-diff/{id}` every 30s.
**Impact:** Arbitrary command execution on the event-handler host process (holds Docker socket, GH_TOKEN, AUTH_SECRET, SQLite DB) — a container-to-host escape. Trigger conditions are routine (any untracked file with metacharacters in its name + any diff refresh) and reachable by the (untrusted) coding agent without a malicious user.
**Recommendation:** Never pass git-sourced filenames through a shell. Use `git ls-files -z`, split on NUL, and count lines with Node `fs` (or `execFileSync('grep', ['-c', '', file], opts)` with `shell:false`). Apply the same fix wherever git-listed paths are interpolated.

### [P0] Client-controlled `scope` path traversal sets agent cwd outside the workspace (ai-llm, chat-server) — lib/ai/scope.js:12
**Status:** confirmed
**Category:** security
**Description:** `resolveAgentScope(repoRoot, scope)` does `path.join(repoRoot, scope)` with no containment/normalize check. `scope` comes from the `/stream/chat` request body (`lib/chat/api.js:16`), forwarded verbatim (`api.js:89`) into `chatStream`→`streamViaSdk` (`lib/ai/index.js:257`)→Claude Agent SDK `options.cwd` (`lib/ai/sdk-adapters/claude-code.js:138`). `scope:'../../../..'` resolves the working dir above the per-workspace clone (to the volume-mounted `/app` containing `.env`, `data/thepopebot.sqlite`, other workspaces). The SDK runs **in-process** inside the event-handler container, so this is host-side. `/stream/chat` only requires any authenticated session (`lib/chat/api.js:10`; middleware only gates `/admin`). The same unvalidated `scope` also drives `buildCodingAgentSystemPrompt` (`lib/ai/system-prompt.js:20`), enabling arbitrary `SYSTEM.md` reads into the prompt (P1, below). (Reported as P0 in ai-llm and P1 in chat-server; same root cause.)
**Impact:** An authenticated non-admin escapes the per-chat sandbox: the agent reads (and with `codeModeType:'code'`, writes/executes) arbitrary host files reachable from the event-handler process — the settings DB, `.env`, AUTH_SECRET, GH_TOKEN, LLM keys, OAuth tokens, other users' workspaces. A reachable secret-leak / data-tamper primitive.
**Recommendation:** Validate `scope` before use: reject `..` segments, leading `/`, null bytes, then assert `path.resolve(repoRoot, scope)` stays within `repoRoot`. Better: only accept scopes that exactly match an entry from the legitimate `/chat/scopes` enumeration (treat as a known-key lookup, not a free-form path). Apply the same guard in `buildCodingAgentSystemPrompt`.

### [P1] `codeModeType` is client-controlled, no allowlist, maps to SDK `bypassPermissions` (ai-llm) — lib/ai/sdk-adapters/claude-code.js:146
**Status:** confirmed — **Category:** security
`options.permissionMode = permissionMode === 'plan' ? 'plan' : 'bypassPermissions'`; any value other than `'plan'` grants full write/execute. `permissionMode` is the client `codeModeType` (`lib/chat/api.js:16`, defaulted to `'plan'` only at api.js:86), never validated; the same value drives container `dangerous` mode (`lib/ai/index.js:327`). **Impact:** any authenticated user requests full write/execute via `codeModeType:'code'`; combined with the `scope` traversal finding, turns the escape into host write/RCE. **Recommendation:** allowlist `['plan','code']`, map explicitly so unexpected/blank values fail closed to plan, and consider gating write mode by role/config.

### [P1] Client `scope` causes host-side arbitrary `SYSTEM.md` read into the system prompt (ai-llm) — lib/ai/system-prompt.js:20
**Status:** confirmed — **Category:** security
`buildCodingAgentSystemPrompt('agent', skillsDir, scope)` builds `path.join(PROJECT_ROOT, scope, 'SYSTEM.md')` from the same unvalidated `scope`; with `..` it reads any `SYSTEM.md` on the host and `render_md`-inlines its `{{include}}`s into the LLM prompt. **Impact:** authenticated user exfiltrates arbitrary host `SYSTEM.md` (and transitively included `.md`) via agent behavior. **Recommendation:** apply the same containment/allowlist validation as `scope.js`; assert the path stays within `PROJECT_ROOT/agents`.

### [P1] Missing chat-ownership check in /stream/chat → cross-user chat/workspace/agent hijack (web-app) — lib/ai/index.js:43-100
**Status:** confirmed — **Category:** security
`chatStream(threadId,...)` resolves `getChatById(threadId)` from the client-supplied `chatId` with **no** `existingChat.userId === options.userId` check (`getChatById` has no user filter). For an existing chat it saves the attacker's message into the victim thread, resumes the victim's workspace/agent session, injects the victim's agent-job secrets (`injectSecrets=true`, line 137), runs the agent (write/dangerous if requested) in the victim's repo, and streams output back to the attacker. Every sibling read handler enforces ownership — `/stream/chat` is the gap. **Impact:** cross-user IDOR enabling data disclosure and code execution in another user's repo context (mitigated only by UUID unpredictability). **Recommendation:** after `getChatById`, reject when `existingChat.userId !== options.userId`; apply the same to any client-supplied `workspaceId`.

### [P1] Container lifecycle server actions lack admin/ownership check (chat-server) — lib/chat/actions.js:575-619
**Status:** confirmed — **Category:** security
`stopDockerContainer` (575), `startDockerContainer` (592), `removeDockerContainer` (609) call only `requireAuth()` then pass caller-supplied `containerName` straight to Docker (`removeContainer` uses `DELETE .../{name}?force=true`). No `requireAdmin()`, no ownership check; non-admin accounts exist. **Impact:** any logged-in non-admin can stop/force-remove the event handler, the litellm sidecar, or other users' agent-job containers (DoS / cross-tenant sabotage); names are enumerable via the also-ungated `/stream/containers`. **Recommendation:** `requireAdmin()` or resolve the container to a `code_workspace` row and verify `ws.userId === user.id`; also gate the container streams/pages on admin.

### [P1] OAuth client secret leaked to any authenticated user (chat-server, chat-ui) — lib/chat/actions.js:1521-1530
**Status:** confirmed — **Category:** security
`getOAuthSecretCredentials()` is gated by `requireAuth()` (any user) and returns `getAgentJobSecretOAuthCredentials(name)` → decrypted `{ clientId, clientSecret, tokenUrl }` in **plaintext**. Every other secret read in the file masks; the write paths use `requireAdmin()`. The consumer `AddSecretDialog` (`settings-jobs-page.jsx:231-240`) loads it into React state, and the dialog is reachable from `chat-input.jsx` (KeyIcon, gated only by `!codeMode`) **outside** `/admin`, so the middleware admin gate never applies; `getAgentJobSecrets()` is likewise `requireAuth`-only. **Impact:** non-admin enumerates agent-job secret names and reads OAuth `clientId`/`clientSecret` (long-lived provider credentials) in the browser. **Recommendation:** raise `getOAuthSecretCredentials` and `getAgentJobSecrets` to `requireAdmin()`, gate the chat-input KeyIcon on admin role, and never return `clientSecret` to the client (pre-fill `clientId` + masked placeholder; reuse the stored secret server-side).

### [P1] Container SSE endpoints: any authenticated user enumerates all containers, tails any log, and force-removes any container (containers-docker, web-app) — lib/containers/logs.js:25-44 / lib/containers/stream.js:4-39
**Status:** confirmed — **Category:** security
`/stream/containers` and `/stream/containers/logs` check only `session?.user`. `stream.js` returns every network container (names/images/state/CPU/mem). `logs.js` takes `name` verbatim (no ownership/allowlist) into inspect/tail, and `cleanup=true` calls `removeContainer(name)` (`?force=true`) on every terminal path. The sibling cluster stream correctly enforces `cluster.userId === session.user.id`; these do not, and `/stream/*`/`/containers` are not under the `/admin` middleware gate. (Multiple overlapping reports merged.) **Impact:** cross-tenant disclosure (logs may contain prompts, repo paths, secrets in traces) + DoS/data-loss (force-remove the event handler, litellm, or other users' in-flight jobs); enumeration feeds the names needed for removal. **Recommendation:** `requireAdmin()` for both endpoints, or resolve container→owning workspace/cluster and verify `userId === session.user.id`; gate `cleanup=true` behind that check; consider redacting log lines.

### [P1] Cluster webhook accepts any valid API key (incl. agent-job keys) to trigger any user's cluster role (cluster) — lib/cluster/runtime.js:162-191
**Status:** confirmed — **Category:** security
`handleClusterWebhook` (POST `/api/cluster/:clusterId/role/:roleId/webhook`) is gated only by `checkAuth()`/`verifyApiKey()`, which accepts both long-lived `api_key` and per-container `agent_job_api_key` rows. It verifies role↔cluster but never ties the caller's key to the cluster's owner, and there is no per-role webhook secret. **Impact:** cross-tenant privilege escalation — any key holder (including a prompt-injected agent-job container carrying `AGENT_JOB_TOKEN`) spins up cluster workers it doesn't own, which launch with the instance's `CLAUDE_CODE_OAUTH_TOKEN`/`GH_TOKEN` and run attacker-supplied `payload.prompt`. **Recommendation:** bind webhooks to the owning user (match key `createdBy` to cluster owner) or add a per-role timing-safe secret; at minimum reject `agent_job_api_key` on this route.

### [P1] Path traversal in `getSessionLog` via unsanitized `roleShortId`/`sessionName` (cluster) — lib/cluster/actions.js:336-360
**Status:** confirmed — **Category:** security (reported 3×; merged)
`getSessionLog` builds `path.join(clusterDir(cluster), 'logs', 'role-'+roleShortId, sessionName, filename)`; only `filename` is allowlisted. `roleShortId`/`sessionName` arrive from the browser server-action with no canonicalization or base-dir containment, so `..` segments escape into other clusters' log trees (or any same-named file the process can read). **Impact:** authenticated cross-tenant disclosure of other users' session logs (prompts/outputs/trigger payloads) despite the cluster ownership check. **Recommendation:** `path.resolve` and assert `startsWith(logsBase + path.sep)`; validate `roleShortId` against `/^[a-f0-9]{8}$/` and `sessionName` against the known timestamp pattern. (`getWorkerPrompts`, actions.js:257-282, has the analogous unvalidated suffix-match pattern — P3.)

### [P1] Forwarded container ports exposed via Traefik with no authentication or ownership check (code-workspaces) — lib/code/port-forwards.js:111-166
**Status:** confirmed — **Category:** security
`forwardPort()` writes a Traefik route mapping `${workspaceId.slice(0,8)}-${port}.${SSL_DOMAIN}` → `http://{containerName}:{port}` with **no** middleware (no ForwardAuth/basicAuth). Unlike every server action and the ws-proxy, the subdomain is reachable by anyone who can route to the host; the host is only 8 hex chars + port (guessable), and with `SSL_DOMAIN` the wildcard is publicly resolvable. **Impact:** any forwarded service (dev servers, admin UIs, DBs over HTTP) is internet-reachable with no auth, bypassing the session/ownership model. **Recommendation:** attach a ForwardAuth middleware verifying session + workspace ownership to every `wksp-*` router, or require an unguessable token; default to localhost-only.

### [P1] Unvalidated `workspace.branch` flows unquoted into host-side `execSync` git commands (code-workspaces) — lib/code/actions.js:592-603
**Status:** confirmed — **Category:** security
In `getWorkspaceDiffStats`/`getWorkspaceDiffFull`, `baseBranch` (= `workspace.branch`) is interpolated unquoted into host `execSync` git commands. `workspace.branch` is written verbatim by `updateWorkspaceBranchHandler`→`updateBranch` (`lib/db/code-workspaces.js:111`) with no charset validation. **Impact:** an authenticated owner sets a branch like `` main; <cmd> `` (or backticks) → command execution on the event-handler host (which has Docker-socket access). Self-scoped but crosses the container/host boundary. **Recommendation:** validate branch names against a strict git-ref allowlist at the write boundary, and use `execFileSync` with arg arrays instead of string interpolation.

### [P1] Single global encryption key with hardcoded static salt and no rotation/versioning (db) — lib/db/crypto.js:7-24,31-42
**Status:** confirmed — **Category:** security
All AES-256-GCM secret encryption derives one process-wide key via `pbkdf2Sync(AUTH_SECRET, SALT, 100000, 32, 'sha256')` with `SALT = 'thepopebot-config-v1'` (a public package constant). Every encrypted `settings` row (GitHub/LLM/Telegram tokens, webhook secrets, OAuth tokens, custom provider configs, agent-job secrets) uses this one key; no per-record salt, no key-version in the `{iv,ciphertext,tag}` envelope, no rotation path. **Impact:** anyone who obtains `AUTH_SECRET` (env dump, leaked `.env`, backup) decrypts the entire store offline in one step; rotating `AUTH_SECRET` silently makes every secret undecryptable (`decrypt()` returns null), mass-degrading webhook auth and agent secret injection with no detection. **Recommendation:** use a random per-record salt stored in the envelope and a key/version identifier to enable rotation; store `AUTH_SECRET` outside the DB host; derive the salt from a stored random value created at first init rather than a package constant. (See also the related P3 hardcoded-salt finding in config-misc.)

### [P1] Stale `__Secure-` session cookie cleanup fails over HTTPS (auth) — lib/auth/middleware.js:40-44
**Status:** confirmed — **Category:** correctness
Stale cookie deletion calls `response.cookies.set(name, '', { maxAge: 0, path: '/' })` with no `secure: true`. Per the cookie-prefix spec, browsers reject a `Set-Cookie` for a `__Secure-`-prefixed name that lacks `Secure`, so the production cookie (`__Secure-authjs.session-token*`, used behind Traefik/HTTPS) is never cleared. **Impact:** after `AUTH_SECRET` rotation or container restart, a user with an undecryptable `__Secure-` cookie hits an infinite redirect loop `/`→`/login` — the feature is a no-op exactly where it matters (HTTPS/production). **Recommendation:** include matching attributes when clearing prefixed cookies: `{ maxAge:0, path:'/', secure:true, httpOnly:true, sameSite:'lax' }`. (Reported again at P3.)

### [P1] `setupAdmin()` is an unauthenticated, unthrottled first-admin creation endpoint (auth) — lib/auth/actions.js:26
**Status:** needs-review (confidence: medium) — **Category:** security
`setupAdmin(email,password)` is a session-less `'use server'` action (must be, to bootstrap); it only checks `password.length >= 8`. No rate limiting, IP gating, or one-time setup token. The atomic `createFirstUser` transaction prevents a second admin once one exists, so the window is "deployed but setup not yet completed." **Impact:** if the instance is network-reachable before the operator finishes setup, the first caller becomes the permanent admin and locks out the operator. **Recommendation:** require a setup token (printed to logs / env on first boot), restrict to loopback until completed, and add rate limiting; document that the instance must not be exposed pre-setup.

### [P1] Webhook action `fetch()` target URL fully config-controlled, no allowlist (SSRF) (actions-scheduling, tools-integrations) — lib/actions.js:21-33
**Status:** confirmed — **Category:** security
`type:'webhook'` issues `fetch(action.url, fetchOpts)` with `action.url`/`method`/`headers` taken verbatim from CRONS.json/TRIGGERS.json — no scheme allowlist, no IP/host denylist (link-local/metadata/RFC1918/loopback), no redirect limit; body forwards the raw inbound request data. **Impact:** SSRF to cloud metadata (`169.254.169.254`), internal services, the Docker socket proxy, plus exfiltration of request bodies — unauthenticated when chained with the public-route trigger bug. **Recommendation:** require an https scheme allowlist, reject loopback/link-local/RFC1918/metadata ranges (re-validate after redirects or `redirect:'manual'`), and avoid auto-forwarding the full body unless opted in.

### [P1] SSRF via unvalidated OAuth `tokenUrl` in token refresh, reachable by any agent_job_api_key (api-external) — api/index.js:134-143
**Status:** needs-review (confidence: high) — **Category:** security
`handleGetAgentSecret()` reads `freshParsed.tokenUrl` from a stored secret and passes it to `refreshOAuthToken()`→ unconstrained `fetch(tokenUrl, { method:'POST', body: client_id/secret/refresh_token })` (`lib/oauth/helper.js:74`). No scheme check, no private/metadata-range block. Triggerable by any holder of an `agent_job_api_key`. The same pattern exists in `handleOAuthCallback`→`exchangeCodeForToken`. **Impact:** SSRF to internal/metadata endpoints + leakage of `client_id`/`client_secret`/`refresh_token` to the chosen URL. **Recommendation:** validate `tokenUrl` before every fetch (https-only, reject private/loopback/link-local ranges, provider-endpoint allowlist) and at save time. (Admin-initiated variant tracked as P2 in config-misc.)

### [P1] `loadCrons()` unguarded `JSON.parse` at startup aborts entire server boot (actions-scheduling) — lib/cron.js:247
**Status:** confirmed — **Category:** reliability
`loadCrons()` does `JSON.parse(fs.readFileSync(cronFile))` with no try/catch and then `for...of` over the parsed value; `config/instrumentation.js:56` calls it in `register()` with no surrounding try/catch. The sibling `reloadCrons()` pre-validates JSON and bails — the cold-start path does not. **Impact:** one malformed (or non-array) `CRONS.json` prevents the event handler from booting — version checks, cluster runtime, maintenance cleanup, and config watchers never initialize. **Recommendation:** wrap read+parse in try/catch mirroring `reloadCrons` (log, set `_cronTasks=[]`, return), guard against non-array, and wrap each `register()` init call so one subsystem's failure doesn't block others.

### [P1] Headless agent container leaks (never removed) when the chat stream consumer disconnects early (ai-llm) — lib/ai/index.js:425-461
**Status:** confirmed — **Category:** reliability
`streamViaContainer()` only stops/removes the container after the `for await ... parseHeadlessStream` loop completes; the `finally` (456-460) only flushes `pendingText` — it never calls `removeContainer` or destroys the Docker log stream. On early consumer abandonment (client disconnect / navigation), the generator's `.return()` runs only that `finally`. `runHeadlessContainer()` launches without `AutoRemove` (`docker.js:470-491`), so the container keeps running (LLM spend, CPU, bind mount), then lingers `Exited` forever; the follow-mode log stream leaks a handle. **Impact:** under common streaming-chat disconnects, every abandoned turn leaks a container + Docker-socket stream, accumulating until host resources are exhausted. **Recommendation:** wrap the body in try/finally that best-effort destroys the log stream and removes the container on any exit path (tolerating 404), and consider `AutoRemove:true` for headless containers as defense-in-depth.

### [P1] `code`-mode system prompt read from empty template path; populated template ignored (ai-llm) — lib/ai/system-prompt.js:29
**Status:** confirmed — **Category:** correctness
For code mode the function reads `coding-workspace/SYSTEM.md`, whose scaffolded template is 0 bytes, so `render_md` returns '' and the function returns null. The real populated prompt lives in `event-handler/code-chat/SYSTEM.md` (the path docs reference). Agent mode reads `agent-job/SYSTEM.md` rather than the documented `event-handler/agent-chat/SYSTEM.md`. **Impact:** default installs run code-mode chats with no custom system prompt; editing the documented file has zero effect; no error is thrown. **Recommendation:** read `event-handler/code-chat/SYSTEM.md` (code) and `event-handler/agent-chat/SYSTEM.md` (agent) with the scoped override preserved; remove the stale empty template or reconcile docs to a single source of truth.

### [P1] Long formatted Telegram responses can split mid-HTML-tag and fail to send (channels) — lib/tools/telegram.js:236-249,180-211
**Status:** confirmed — **Category:** reliability
`sendMessage()` converts markdown→HTML first, then `smartSplit(text, 4096)` splits purely by length/whitespace with no tag awareness, sending each chunk `parse_mode:'HTML'`. Splits land mid-tag/mid-entity → Telegram 400 → grammY throws. The flush calls at telegram.js:163 and :190 are unguarded, aborting the turn. **Impact:** any LLM reply or job DM > ~4096 chars containing formatting (the common case) can throw, truncating the response or silently dropping a DM. **Recommendation:** split before HTML conversion (convert each chunk independently) or make the splitter HTML-aware (rebalance tags at boundaries); at minimum retry a failed chunk as plain text.

### [P1] `scheduleWorkerDirCleanup`: unhandled rejection + unbounded interval leak, and premature deletion of live worker dir on Docker error (cluster, containers-docker) — lib/cluster/execute.js:280-295
**Status:** confirmed — **Category:** reliability (two merged reports, same code)
The async `setInterval` callback `await listContainers(containerName)` has no try/catch, so any rejection is an unhandled rejection every 5s, and `clearInterval` only runs in the `if (!still)` branch — a persistent Docker error loops forever, leaking a timer per cleanup-enabled worker. Separately, `listContainers` returns `[]` on transient non-200, so `still` is undefined while the container is still alive → `fs.rmSync(dirPath, {recursive,force})` deletes the live bind-mounted workspace. **Impact:** unhandled rejections (possible process crash), leaked timers, and corruption/loss of an in-progress agent's filesystem on a transient Docker hiccup. **Recommendation:** wrap the body in try/catch; distinguish "absent" from "query failed" (treat error as still-running, skip the tick); add a max-attempts/max-lifetime cap; only delete on a successful query that returned no match.

### [P1] `verifyApiKey`/`backfillLastUsedAt` JSON.parse settings rows without guarding malformed values (db) — lib/db/api-keys.js:135-138
**Status:** needs-review (confidence: medium) — **Category:** reliability
`verifyApiKey()` iterates all key rows doing `JSON.parse(row.value)` then `Buffer.from(parsed.key_hash,'hex')` with no try/catch; a malformed value or missing `key_hash` (→ `Buffer.from(undefined)` throws) aborts the whole loop — the only auth check — failing all valid keys. `backfillLastUsedAt()` has the same unguarded parse at startup, aborting after the first bad row. **Impact:** one corrupted key row turns every API request into a 401/500 (DoS for all external callers and agent-job secret fetches) and stops the backfill mid-run. **Recommendation:** wrap the per-row parse in try/catch (continue on failure) and skip rows missing `key_hash`.

### P2 findings

The following P2 issues are grouped by theme; each is confirmed unless noted. Cite file:line as given.

**Webhook-secret timing-safety (security, convention) — recurring.** The Telegram webhook secret is compared with non-constant-time `!==` at `lib/channels/telegram.js:36`, while the sibling GitHub path uses `safeCompare`/`timingSafeEqual` (`api/index.js:33-39,345`) and API keys use `timingSafeEqual` (`lib/db/api-keys.js:138`). This is the same finding reported many times across `api-external`, `auth`, `channels`, and `tools-integrations`. *Recommendation:* factor `safeCompare` into a shared util and use it in `telegram.js receive()`.

**Agent-job secret scoping (security) — api/index.js:98-156.** Any `agent_job_api_key` can read/list **every** agent-job secret (flat store, no per-job ACL); `handleGetAgentSecret`/`handleListAgentSecrets` gate only on key *type*. A single compromised agent-job container exfiltrates all jobs' OAuth client secrets, refresh tokens, and plaintext env secrets. *Recommendation:* associate each key with its job/owner and restrict get/list to owned secrets. (Related convention issue: these handlers re-`verifyApiKey` and deref `record.type` with no null guard — api/index.js:99-100, also P3.)

**SSRF via custom LLM provider `baseUrl` (security) — lib/ai/helper-llm.js:55.** `createOpenAICompatible({ baseURL: custom.baseUrl, apiKey: custom.apiKey })` uses an unvalidated stored URL for every helper-LLM call (autoTitle/summaries); a `baseUrl` of `169.254.169.254`/`litellm:4000` turns routine title generation into internal SSRF with the apiKey leaked. Admin-only to configure → P2. *Recommendation:* validate at save/use (https, reject private/link-local/metadata, no URL creds).

**LiteLLM YAML injection (security) — lib/chat/actions.js:1259-1267.** `syncLitellmConfig()` interpolates `p.baseUrl`/`p.apiKey` into hand-built YAML with no quote/newline escaping; a crafted value injects arbitrary LiteLLM directives into the shared proxy config. Admin-only. *Recommendation:* validate URL/reject control chars, or serialize with `js-yaml`.

**`getWorkspace` ownership gap (security) — lib/chat/actions.js:483-494.** `getWorkspace(id)` returns repo/branch/containerName/featureBranch with `requireAuth()` but no `ws.userId` check; the container name then feeds the unguarded stop/remove actions. *Recommendation:* add the ownership comparison like its siblings.

**`finalizeChat` missing ownership check (security/correctness) — lib/chat/api.js:658-684.** Reported 4×. Reads `chatId` from the body and calls `autoTitle(chatId, message)` before any ownership check, then returns `codeWorkspaceId`/`featureBranch`. *Impact:* cross-user IDOR — overwrite another user's "New Chat" title with attacker text (burning a helper-LLM call) and read its workspace/branch. *Recommendation:* `getChatById` first, return 403/404 unless `chat.userId === session.user.id`.

**Markdown `{{include}}` path traversal (security) — lib/utils/render-md.js:117-122.** Reported 2×. `path.resolve(PROJECT_ROOT, includePath)` with no containment check; a `{{../../x.md}}` token reads any `*.md` on the host into the agent system prompt. Admin-authored config today → defense-in-depth. *Recommendation:* assert `path.relative(PROJECT_ROOT, resolved)` is not `..`/absolute before reading.

**OAuth `tokenUrl` admin-supplied (security) — lib/oauth/helper.js:31-44** and **OAuth state has no nonce/expiry/session binding — lib/oauth/helper.js:10-14.** Admin-initiated SSRF to an arbitrary token endpoint with the client secret in the body; and a replayable, non-CSRF-bound state blob. *Recommendation:* allowlist `tokenUrl` against `OAUTH_PROVIDERS` (https, no private ranges); add a random nonce + `iat`, reject expired/foreign states.

**Config/secret robustness (reliability) — db.** `decrypt()` throws on malformed/tampered/old-key ciphertext (`lib/db/crypto.js:49-58`); `getConfigSecret()` swallows all decrypt failures and returns null, silently disabling secrets after an `AUTH_SECRET` change (`lib/db/config.js:73-86`). *Recommendation:* validate envelope fields, log on failure, distinguish "absent" from "undecryptable," and degrade gracefully per-secret.

**Non-transactional cascade deletes (reliability) — lib/db/chats.js:97-101 (and clusters.js:88-92).** `deleteChat`/`deleteCluster` run two un-transacted deletes with no FKs; a crash between them orphans rows. `createFirstUser` already uses `db.transaction()`. *Recommendation:* wrap each cascade in a transaction or declare FK cascades.

**`deleteApiKeyById`/`deleteOAuthTokenById` delete by id with no type guard (security/reliability) — lib/db/api-keys.js:101-104 (and oauth-tokens.js:90-93).** Reported 2×. The single multiplexed `settings` table means a wrong-type id deletes a webhook secret/provider config instead of the intended row. *Recommendation:* add `and(eq(settings.id,id), eq(settings.type,'api_key'))` (and `'config_secret'`).

**Migration/key-list drift (reliability) — lib/db/config.js:366-384.** `MIGRATE_SECRETS`/`MIGRATE_CONFIG` are a duplicated copy of `SECRET_KEYS`/`CONFIG_KEYS` and have already diverged (DeepSeek/MiniMax/Mistral/xAI/OpenRouter/NVIDIA/AssemblyAI keys, the whole `CODING_AGENT_*` family). Legacy `.env` values silently never migrate. *Recommendation:* derive the migrate lists from the single source of truth.

**Telegram verify-code weakness (security) — lib/db/user-channels.js:6-15,93-117 (and channels reports).** 32-bit code (`randomBytes(4)`→8 hex), 10-min TTL, no attempt counting/lockout, distinct error strings (`verify.js:13-17`) form an enumeration oracle; the `/verify` path is pre-auth and unthrottled. *Impact:* online brute-force / any leaked code binds an attacker's chat to a victim account. *Recommendation:* ≥16 random bytes, per-chat/IP rate limiting + lockout, generic failure message.

**`resolveHostPath` poisoned cache (reliability) — lib/tools/docker.js:745-771.** Reported 3×. On a failed first self-inspect, `_hostMounts` stays `[]` forever and the function returns the container-internal path as the host bind source, silently producing wrong/empty workspace mounts for the process lifetime. *Recommendation:* only memoize on a successful inspect; log/throw on fall-through while in Docker.

**Per-container agent-job key lifecycle (security/reliability) — lib/tools/docker.js:243-245,463-465,930-932.** A key row is created per launch but never deleted on container removal — only the hourly time-based cron reaps it (≤24h), so a leaked token stays valid long after the container exits. Plus the SDK-vs-container classification keys off fragile `key.includes('sdk')` (`lib/maintenance.js:19-20`, reported 2×), risking deletion of a live container's key after 1h. *Recommendation:* delete the key on intentional removal; store an explicit discriminator instead of substring-matching.

**Action/cron schema validation missing (reliability) — lib/actions.js:13-44 / lib/cron.js:247-261 (reported 3×).** No validation that `type ∈ {agent,command,webhook}` or that required fields exist; a typo'd `type` silently defaults to `'agent'` and spawns Docker jobs. *Recommendation:* validate at load time (zod), reject unknown types, log+skip invalid entries.

**Webhook/version-check fetch has no timeout (reliability) — lib/actions.js:32 (and cron.js).** No `AbortController`; a hung URL leaves the promise pending for the OS socket timeout. *Recommendation:* `AbortSignal.timeout(15000)`.

**Unbounded buffering / unbounded persisted output (reliability) — lib/ai/headless-stream.js:55-89; lib/ai/line-mappers.js:36.** `stderrChunks` retains every stderr frame for the whole run (cap applied only after the single joined chunk); unparsed stdout lines are persisted to the chat DB with no size cap. *Impact:* memory growth / OOM on a noisy agent; storage exhaustion + stored untrusted content. *Recommendation:* keep only a tail at the source; cap persisted text per turn.

**PDF attachments sent as image blocks (correctness) — lib/ai/sdk-adapters/claude-code.js:157-166.** `application/pdf` is classified `image` and emitted as an `image` block with `media_type:'application/pdf'`, which the Anthropic API rejects. *Recommendation:* emit a `document` block for PDFs.

**Secret leakage into LLM prompt/logs (security) — lib/ai/index.js:515.** `summarizeAgentJob()` forwards the raw agent-job log (which can contain injected secrets/env) into the helper-LLM prompt and logs a 200-char result slice. *Recommendation:* redact known secret patterns before composing the prompt; reduce logging.

**`LLM_MAX_TOKENS` config dead (convention) — lib/ai/helper-llm.js:75.** Documented default ignored; helper functions use hardcoded per-call values and never read `getConfig('LLM_MAX_TOKENS')`. *Recommendation:* fall back to the configured value or drop the claim/key.

**Thinking pseudo-tool block never closed on early termination (reliability) — lib/chat/api.js:180-216,269-275.** If the stream ends after `thinking-start`/`thinking` but before `thinking-end`, the client renders a perpetually-loading thinking block. *Recommendation:* emit a closing `tool-output-available` after the loop and in error/auto-run branches.

**Auth/role robustness (security/reliability/correctness) — auth.** `updateProfile()` looks the user up by mutable JWT email, breaking re-verification after an admin email change (`lib/auth/actions.js:161-169`); no role allowlist in `addUser`/`editUser`/`createUser` (`lib/auth/actions.js:53-104`, `lib/db/users.js:35-52`); JWT role/deletion staleness means demoted/deleted users keep privileges until token expiry (`lib/auth/edge-config.js:13`). *Recommendations:* look up by immutable `id`; validate role against `['user','admin']`; re-validate role against the DB for privileged ops (or a `tokenVersion`).

**Telegram message handling (reliability/correctness) — channels.** Unguarded text-flush `sendMessage` aborts the turn and abandons the agent stream (`telegram.js:162-191`); tool-result status emoji keys off a naive substring `'error'` match (`telegram.js:178-181`); placeholder restore uses single-occurrence string replace vulnerable to token desync (`telegram.js:83-88,133-135`); voice transcription returning empty silently drops the message (`telegram.js:60-87`); document attachments forwarded with attacker-supplied MIME and no size cap (`telegram.js:103-112`). *Recommendations:* guard flushes + drain the iterator; carry an explicit error flag; use a collision-resistant placeholder; surface empty-transcription; enforce size/type allowlist.

## Cross-Cutting Themes

1. **Untrusted input into shells (systemic injection).** The same anti-pattern — string-interpolating attacker- or config-influenced values into `child_process.exec`/`execSync`/`sh -c` — appears in the action executor (`lib/actions.js`), trigger templating (`lib/triggers.js`), host-side git diffs (`lib/code/actions.js`), in-container file ops (`lib/code/actions.js`, `lib/tools/docker.js execInContainer`), and across the CLI/setup (`bin/cli.js`, `setup/*.mjs`). Root cause: no project-wide rule to use argv-array `execFile`/`spawn` with `shell:false`. **Fix once, everywhere.**

2. **Trigger fan-out runs before authentication.** The single ordering decision in `api/index.js` (fire triggers on every POST, before the route's own secret check) is what elevates the injection/SSRF sinks to *unauthenticated* RCE. It is the connective tissue between the P0s.

3. **Authorization enforced inconsistently across the user/admin/tenant boundary.** Reads use `requireAuth` while sibling writes use `requireAdmin`; some chat handlers check `chat.userId` and others (`/stream/chat`, `finalizeChat`, `getWorkspace`) don't; container/cluster-log streams omit the ownership check their cluster-log sibling performs; the cluster webhook ignores key ownership. The pattern exists in the codebase but is applied ad hoc, leaving repeated IDOR/cross-tenant gaps.

4. **Path traversal from client/config strings to the filesystem.** `scope` (cwd + system prompt), `getSessionLog`/`getWorkerPrompts` (cluster logs), `file_watch.paths` (chokidar), and `{{include}}` (render_md) all `path.join`/`resolve` untrusted segments with no `startsWith(base)` containment assertion.

5. **SSRF wherever a URL comes from config/state.** Webhook actions, custom-provider `baseUrl`, and OAuth `tokenUrl` all `fetch()` user/config-supplied URLs with no scheme allowlist or private/metadata-range block.

6. **Single-key crypto + fragile secret lifecycle.** One global key with a public salt, no rotation/versioning, swallowed decrypt errors, per-container keys never proactively revoked, and substring-based key classification together make `AUTH_SECRET` a single point of total compromise and make rotation a silent footgun.

7. **Unguarded `JSON.parse`/`decrypt`/Docker calls cause fail-closed-but-total outages.** Cold-start cron load, `verifyApiKey`, cluster trigger load, and config reads all throw on a single bad row/response, taking down boot, all auth, or all cluster triggers rather than isolating the failure.

8. **Resource/handle leaks on the unhappy path.** Headless containers, Docker log streams, cleanup intervals, keepalive timers, and `_locks` entries all leak on early disconnect/error because cleanup lives only on the success path.

9. **Doc/code and dual-implementation drift.** Multiple CLAUDE.md files document removed/renamed APIs; reads bypass the documented route-handler rule; `init` vs `sync` scaffolding, GitHub fetch wrappers, Telegram `setWebhook` variants, and migrate-key lists are forked copies that have already diverged.

## Notes & Limitations

- This was a **static review of a non-git, untested snapshot** of the published `thepopebot` v1.2.82 source. There are no tests and no commit history to diff against; findings are based on reading the actual files in scope. Dynamic exploitability (especially network-timing attacks, race windows, and platform-specific `execSync`/`copyFileSync` behavior) was not exercised except where the underlying report explicitly noted a local reproduction.
- **Severity reflects worst realistic case** given the documented deployment (Next.js event handler behind Traefik/HTTPS, Docker socket mounted, multi-user with non-admin accounts, agent-job containers running untrusted LLM-driven tasks). In a single-operator, non-internet-exposed deployment, several P0/P1 cross-tenant and public-route issues are materially lower risk — but the package is published for multi-user use, so they are rated accordingly.
- **`needs-review` items warrant human confirmation** before action: notably the unauthenticated-`setupAdmin` window (depends on deployment exposure timing), the OAuth `tokenUrl` SSRF reachability via agent-job keys, the chunked-cookie WS-proxy auth failure, the `init` symlink crash (platform-dependent `copyFileSync` semantics), the malformed-JSON-version sort, and the raw-DDL-in-`initDatabase` migration-rule violation.
- **Duplicates were merged** where multiple subsystem reports pointed at the same root cause (the public-route trigger ordering, the command-injection sink, the `scope` traversal, the container-SSE auth gap, `getSessionLog`, `resolveHostPath`, the Telegram timing-safe comparison, the agent-job-key classification, and `finalizeChat`). Counts in the Executive Summary are pre-merge totals from the verified-findings input; the Findings section presents the de-duplicated set.
- No new findings were introduced; this report only organizes, deduplicates, and prioritizes the provided verified findings and architecture data.
